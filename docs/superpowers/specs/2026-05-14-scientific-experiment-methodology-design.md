# Scientific Experiment Methodology Design

## Status

This design supersedes and merges the useful parts of these prior design docs:

- `2026-05-13-autonomy-symphony-design.md`
- `2026-05-13-b200-profiled-experimentation-design.md`
- `2026-05-13-b200-real-probe-run-design.md`
- `2026-05-13-temporal-torch-experimentation-trial-runs-design.md`

The new direction keeps their best ideas: Temporal discipline, worktree and
artifact isolation, profile-aware probing, trial observability, failure
classification, cancellation behavior, and reporting. It drops competing
concepts that make the system harder to maintain: org trackers, arbitrary task
automation, TSV files as the state source, and Qwen/B200 assumptions inside
generic infrastructure.

## Purpose

Build one repo-level methodology for conducting scientific experiments. The
methodology should be generic. Qwen-VL fine-tuning should consume it through a
domain adapter instead of owning a separate orchestration framework.

The system has two high-level workflows:

- `ProbeWorkflow`: generate candidate training intent configs, run them one by
  one, collect results, and mark the best intents as selected.
- `TrialWorkflow`: run one experiment trial from an intent, including preflight,
  execution, heartbeat progress, analysis, and report generation.

## Architecture

Create one top-level package:

```text
research/
  db.py
  workflows.py
  activities.py
  manager.py
  runners.py
  preflight.py
  artifacts.py
  reports.py
  cli.py
  temporal.py
```

`research/` owns generic experiment methodology:

- SQLite schema and state transitions
- Temporal workflows and activities
- local Temporal setup helpers
- experiment manager CLI
- preflight, worktree, and artifact contracts
- process supervision and cancellation
- heartbeat plumbing
- generic reports

Qwen-VL owns only domain-specific experiment behavior:

```text
qwen-vl-finetune/
  experiments/
    qwen_adapter.py
    profiles/
    launchers/
```

The dependency direction is:

```text
qwen-vl-finetune adapter -> research methodology
```

Generic `research/` code must not depend on Qwen-VL modules, model names,
dataset names, hardware profiles, or trainer log formats.

## Core Concepts

Keep the vocabulary small:

- `Intent`: a desired training configuration generated by probe or authored by a
  user.
- `Experiment`: a durable database record tying an intent to an adapter, state,
  priority, and artifacts.
- `Trial`: one Temporal execution of an experiment.
- `Report`: the human-readable and machine-readable result of a probe or trial.

`autonomy` concepts such as org trackers, done gates, arbitrary code executors,
and generic task queues are out of scope unless they directly support
scientific experiments.

## Database

SQLite is the durable experiment inventory and query surface. Temporal owns
execution history. Artifact directories own large files.

Initial tables:

```text
intents
  id
  adapter
  model
  profile
  phase
  name
  config_json
  objective_json
  source
  status
  score_json
  created_at

experiments
  id
  intent_id
  adapter
  status
  priority
  temporal_workflow_id
  artifact_root
  artifact_subdir
  created_at
  updated_at

trial_runs
  id
  experiment_id
  attempt
  status
  started_at
  finished_at
  heartbeat_json
  metrics_json
  failure_json
  report_path

artifacts
  id
  trial_run_id
  kind
  uri
  size_bytes
  metadata_json
```

Allowed state transitions:

```text
intent: candidate -> selected | rejected | archived
experiment: queued -> running -> succeeded | failed | cancelled
trial_run: preflight -> running -> reporting -> succeeded | failed | cancelled
```

SQLite stores compact state and references. Logs, trainer events, checkpoints,
and reports live under artifact directories and are referenced from the
database by relative paths. TSV exports are allowed for compatibility but are
not a source of truth.

## Workflows

### ProbeWorkflow

`ProbeWorkflow` is a Temporal workflow. It:

1. Calls the adapter to generate candidate intents for a model, profile, and
   objective.
2. Inserts those intents into SQLite with status `candidate`.
3. Creates experiment rows for candidates in probe order.
4. Starts one `TrialWorkflow` at a time.
5. Waits for each trial result.
6. Updates intent scores and statuses from trial reports.
7. Stops when the probe budget is exhausted or enough passing configs exist.
8. Marks the best training intent configs as `selected`.
9. Writes a compact probe report.

Selection is part of probe. There is no separate generic `select` workflow.

### TrialWorkflow

`TrialWorkflow` is a Temporal workflow. It:

1. Reads the experiment and intent from SQLite.
2. Creates a trial run row with status `preflight`.
3. Runs preflight.
4. Invokes the adapter's trial runner activity.
5. Heartbeats compact progress into Temporal and SQLite.
6. Classifies success, failure, cancellation, or timeout.
7. Writes `report.md` and `report.json`.
8. Updates trial, experiment, and artifact rows.

Workflow code stays deterministic and lightweight. File I/O, subprocesses, log
parsing, package checks, and report writing happen in activities.

## Preflight

Generic preflight checks:

- `uv` can execute the project environment.
- The requested adapter can be imported.
- The SQLite database is reachable.
- Worktree and artifact directories can be created.
- The adapter can resolve a trial command.
- Required declared paths are present.

Adapters can add domain checks, such as required datasets, model caches,
launcher files, GPU count, or package versions. Generic code records those
checks but does not understand their model-specific meaning.

## Adapter Boundary

The generic adapter interface stays small:

```python
class ExperimentAdapter(Protocol):
    name: str

    def generate_probe_intents(self, request: ProbeRequest) -> list[Intent]:
        ...

    def preflight(self, intent: Intent, context: TrialContext) -> PreflightResult:
        ...

    def build_trial(self, intent: Intent, context: TrialContext) -> TrialCommand:
        ...

    def parse_progress(self, event_or_log_line: str) -> ProgressUpdate | None:
        ...

    def analyze_result(self, context: TrialContext) -> TrialReport:
        ...
```

Generic infrastructure owns:

- SQLite state transitions
- Temporal orchestration
- artifact directory creation
- worktree setup
- preflight dispatch and recording
- process supervision and cancellation
- heartbeat plumbing
- report persistence
- generic failure envelope: `preflight_failed`, `launcher_failed`, `timeout`,
  `cancelled`, `unknown`

Adapters own:

- model names and model paths
- dataset names and dataset paths
- hardware profiles such as B200, H100, or A100
- probe grid generation
- launch command details
- required `uv` packages and environment checks
- domain metrics such as validation loss, VRAM, and throughput
- domain failure classification such as OOM, NCCL, cache, or dataset errors
- domain report sections

For Qwen-VL, values such as `Qwen/Qwen3-VL-8B-Instruct`,
`visualwebinstruct_train`, `scripts/sft_qwen3_8b.sh`, B200 profile logic, and
trainer log parsing belong in the Qwen adapter. Generic `research/` code may
store them as opaque JSON but must not hard-code them.

## Artifact Contract

Every trial gets an isolated artifact subdirectory:

```text
<artifact-root>/<adapter>/<experiment-id>/attempt-<n>/
  run.log
  events.jsonl
  lifecycle.jsonl
  preflight.json
  system.pre.json
  system.post.json
  heartbeat.jsonl
  report.json
  report.md
```

Adapters may add checkpoints, trainer state, or domain-specific logs under the
same attempt directory or under adapter-declared subdirectories. The database
records artifact references. Generic code should only delete artifacts through
explicit garbage-collection commands scoped to the configured artifact root.

`lifecycle.jsonl` is single-writer supervisor state. It records coarse stages
such as preflight start, command launch, first progress, process exit,
analysis start, and report written. Domain trainers may write their own
structured events to `events.jsonl`, but they do not write lifecycle events.

## CLI and Mise

The Python CLI is the semantic API:

```bash
uv run research db init
uv run research probe --adapter qwen_vl --model <model-ref> --profile <profile>
uv run research manager
uv run research status
uv run research report <experiment-or-probe-id>
```

Use `mise` as the repo-level task runner for routine operation. Top-level
`mise.toml` should expose thin wrappers:

```toml
[tasks.research-db-init]
run = "uv run research db init"

[tasks.research-probe]
run = "uv run research probe --adapter {{arg(name='adapter')}} --model {{arg(name='model')}} --profile {{arg(name='profile')}}"

[tasks.research-manager]
run = "uv run research manager"

[tasks.research-status]
run = "uv run research status"

[tasks.research-report]
run = "uv run research report {{arg(name='id')}}"

[tasks.qwen-probe-b200]
run = "uv run research probe --adapter qwen_vl --model Qwen/Qwen3-VL-8B-Instruct --profile b200"
```

Responsibilities:

- `mise` owns repeatable command invocation.
- `uv` owns Python dependency installation and execution.
- `research` owns experiment semantics.
- Temporal owns orchestration.
- SQLite owns durable experiment state.

Mise tasks must stay thin. They must not duplicate workflow logic, parse
results, manage trial state, or know Qwen internals beyond convenience
defaults.

## Migration Plan

The implementation should migrate to the new shape without maintaining
parallel orchestration systems.

Create:

- top-level `research/` package
- top-level `mise.toml`
- SQLite schema and migration helpers
- generic Temporal helpers, reusing the best of current `local_temporal.py`
- `ProbeWorkflow` and `TrialWorkflow`
- generic preflight, artifact, heartbeat, and report contracts
- Qwen adapter under `qwen-vl-finetune/experiments/`

Merge or remove:

- top-level `autonomy/` package
- old org tracker, done-gate, and task executor methodology
- `qwen-vl-finetune/research/` as a generic orchestration package
- TSV-backed experiment management as the maintained state path
- superseded design docs listed in this document

Temporary compatibility is acceptable only for migration:

- wrappers from old Qwen commands to the new generic CLI
- tests proving old commands delegate or are removed cleanly
- existing artifact data on disk, treated as historical data

## Testing

Unit and integration coverage should focus on the simplified contracts:

- SQLite schema and state transition tests
- adapter contract tests with a fake adapter
- `ProbeWorkflow` dry-run test using fake intents and trial results
- `TrialWorkflow` dry-run test covering preflight, heartbeat, report, and DB
  updates
- Qwen adapter tests for intent generation and command construction
- compatibility tests for any temporary Qwen wrappers
- `mise` smoke test confirming expected tasks exist

No real GPU training belongs in unit tests. Real Qwen probe runs are operational
validation.

## Non-Goals

- Do not build a generic autonomous coding task platform.
- Do not keep org-mode trackers as an experiment state source.
- Do not make TSV files the source of truth.
- Do not hard-code models, datasets, hardware profiles, launchers, or trainer
  metrics in generic `research/` infrastructure.
- Do not support multiple orchestration paths for the same operation.
- Do not require real GPU training in the test suite.
